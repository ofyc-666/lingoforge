# LingoForge 课程作业版 MVP 原子任务拆分

## 执行者规则

默认执行者为 `CODEX`。

`CODEX` 负责架构、Agent、复杂后端、完整前端、Image 2 视觉生成、前后端集成、浏览器验证和最终审查。

`CC_DS` 只允许执行 Codex 已经完整定义的低风险、机械性任务。每个 `CC_DS` 任务必须包含：

- 文件范围；
- 接口；
- 输入输出；
- 实现步骤；
- 测试命令；
- 验收标准；
- 禁止修改范围。

`CC_DS` 完成后，Codex 必须审查和集成。

## 仓库交付标准

最终核心交付物是 GitHub 仓库链接。仓库必须满足：

- 能安装；
- 能启动；
- 能重置 Demo 数据；
- 能运行测试；
- 能走通完整主流程；
- 核心 Prompt 和 Skill 可查看；
- 架构与数据模型可查看；
- `.env.example` 完整；
- README 命令经过实际复跑；
- 不泄露 API 密钥。

每个阶段都应能独立运行或验证。浏览器截图只作为开发核验或 README 可选材料，不再强制固定数量截图、课堂展示或实验报告。

## 阶段 A：项目骨架、配置与健康检查

### A1. 后端 FastAPI 骨架

- 任务ID：`A1`
- 执行者：`CODEX`
- 目标：创建后端最小运行环境、健康检查接口和测试入口。
- 前置依赖：`docs/ARCHITECTURE.md`、`docs/DATA_MODEL.md`、`docs/AGENT_RUNTIME.md` 已通过人工审核。
- 允许修改的文件或目录：`backend/`、`README.md` 中当前可用后端命令部分。
- 禁止修改范围：不实现业务 Workflow、不实现 Agent Runtime、不接真实 DeepSeek、不填充大批种子内容。
- 输入：已确认架构方案、当前环境变量方案、FastAPI 后端职责。
- 输出：FastAPI 应用入口、后端依赖声明、健康检查接口、基础测试配置。
- 逐步实现步骤：
  1. 创建 `backend/app` 与 `backend/tests`；
  2. 定义 FastAPI `app`；
  3. 增加 `GET /health`；
  4. 暴露不含密钥的配置摘要；
  5. 创建后端测试配置。
- 必须执行的命令：
  - `python --version`
  - `python -m pytest backend/tests`
- 测试要求：健康检查测试必须通过；测试不得依赖真实 LLM 或数据库种子内容。
- 验收标准：Windows 本地可运行测试；健康检查返回 `ok`；无业务流程膨胀。
- 失败时回退方式：保留文件结构，回退到只包含 `GET /health` 的最小 FastAPI 应用。
- 完成后的汇报格式：说明新增后端文件、健康检查返回字段、测试命令和结果。

### A2. Vue 前端骨架

- 任务ID：`A2`
- 执行者：`CODEX`
- 目标：创建 Vue 应用、最小 API client 和健康检查页面。
- 前置依赖：`A1`。
- 允许修改的文件或目录：`frontend/`、`README.md` 中当前可用前端命令部分。
- 禁止修改范围：不实现完整业务页面、不引入 Pinia、不调用 Image 2、不实现复杂视觉稿。
- 输入：后端 `GET /health` 返回格式、`VITE_API_BASE_URL` 配置。
- 输出：Vue 应用骨架、健康检查组件、API client、构建配置。
- 逐步实现步骤：
  1. 创建 `frontend` 项目文件；
  2. 创建 `src/api` 健康检查调用；
  3. 创建 `App.vue` 展示后端状态；
  4. 配置 Vite 开发代理或 API base URL；
  5. 保持组件状态简单。
- 必须执行的命令：
  - `node --version`
  - `npm --version`
  - `npm run build`
- 测试要求：前端构建必须通过；页面代码必须真实调用 FastAPI 健康检查。
- 验收标准：Vue 可构建；健康检查 UI 有 loading、success 和 error 状态。
- 失败时回退方式：保留 Vite 最小骨架，回退到仅渲染健康检查请求结果。
- 完成后的汇报格式：说明新增前端文件、API base URL、构建命令和结果。

### A3. 统一配置与 `.env.example`

- 任务ID：`A3`
- 执行者：`CODEX`
- 目标：实现统一配置读取，并正式写入 LLM Provider 环境变量。
- 前置依赖：`A1`。

- 逐步实现步骤：
  1. 定义后端配置读取；
  2. 支持 `LLM_MODE`、`LLM_PROVIDER`、`DEEPSEEK_*`、`DATABASE_PATH`；

- 验收标准：默认 `LLM_MODE=mock`；`.env.example` 完整；健康检查不泄露密钥。
- 失败时回退方式：回退到 stdlib `os.environ` 读取，不引入额外配置库。
- 完成后的汇报格式：说明新增环境变量、默认值、测试结果和密钥保护方式。

## 阶段 B：数据模型、初始化与数据访问

- 任务ID：`B1`
- 执行者：`CODEX`
- 目标：建立 SQLite 连接、可重复初始化和清空机制。
- 前置依赖：`A1`、`A3`。

- 禁止修改范围：不实现业务服务、不填充大批种子内容、不改变已审核的数据边界。

- 输出：数据库连接函数、Schema 初始化函数、清空重建函数、测试。
- 逐步实现步骤：
  1. 使用 Python 标准库 `sqlite3` 建立连接；
  2. 启用外键约束；
  3. 定义 `init_database`；
  4. 定义 `reset_database`；

- 必须执行的命令：
  - `python -m pytest backend/tests`
- 测试要求：使用临时数据库路径；不得污染真实开发数据库。
- 验收标准：数据库可初始化、可清空重建、重复执行稳定。
- 失败时回退方式：移除复杂封装，保留单文件 `schema.sql` + `sqlite3` 初始化。
- 完成后的汇报格式：说明数据库路径、初始化入口、清空入口和测试结果。

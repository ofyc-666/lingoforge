# LingoForge 核心开发 Prompt

## Prompt 1：需求梳理与 MVP 范围设计

请先完整阅读项目根目录下的 `AGENTS.md`、现有 README 和已有需求材料。

本轮先不要编写代码，不要初始化前后端项目，也不要直接决定最终技术栈。请根据当前课程作业要求，使用 `idea-refine`、`spec-driven-development` 和 `interview-me` 等适合的 Skill，帮助我梳理 LingoForge 的产品需求。

项目方向是一个面向英语学习的自适应 Agent，MVP 聚焦 CET-6 词汇与阅读训练。它不能只是普通聊天机器人，也不能只是随机生成英语题目，而应形成以下最小闭环：

1. 收集用户学习目标；
2. 通过短诊断形成初始能力画像；
3. 根据画像和候选词汇制定训练计划；
4. 调用合适的教学 Skill 生成训练；
5. 由确定性程序完成客观题判分；
6. Agent 根据真实答题证据分析错因并提出画像更新建议；
7. 第二轮训练根据更新后的画像发生可解释变化；
8. 使用未参与训练的短隔离测试进行验收。

请重点分析：

* 产品定位和用户真实痛点；
* MVP 必须实现和明确不实现的内容；
* Agent、Skill、固定 Workflow 和确定性程序之间的职责边界；
* 主线训练、机场购票副线和隔离测试之间的数据关系；
* 可验证的用户流程和验收标准；
* 课程作业要求与产品功能之间的映射。

在分析充分、关键问题确认以后，再整理需求文档。不要为了显得完整而扩大范围，不要提前生成大量重复文档。

## Prompt 2：系统与 Agent 架构设计

请先阅读：

* `AGENTS.md`
* 需求文档和 MVP 范围文档
* 用户流程与验收标准
* 当前仓库代码和测试

本轮负责设计 LingoForge 课程作业版 MVP 的系统架构和 Agent 内部架构。

已确认的总体方向是：

* Vue 桌面端 Web 前端；
* FastAPI 后端；
* SQLite 本地数据库；
* 前后端分离运行的模块化单体；
* 单个学习 Agent；
* 使用模型原生 Function Calling；
* 不引入微服务、消息队列、多 Agent、独立向量数据库或不必要的复杂基础设施。

请重点设计以下内容：

1. Vue、FastAPI、SQLite、Agent Runtime 和模型 Provider 的模块关系；
2. Agent、Skill、Function Calling 工具、Workflow 内部服务和确定性程序的职责边界；
3. Agent Runtime 的完整生命周期，包括上下文构建、模型调用、工具调用、结果回填、结构化决策、校验、重试、降级和审计；
4. ContextBuilder 的阶段权限、预算、去重、截断、引用和 Context Manifest；
5. 原始学习证据、用户画像、解释性记忆、副线信号和隔离测试数据的分层保存；
6. 隔离题题干、答案、解析和评分依据在用户作答前不得进入 Agent 上下文；
7. 客观题必须由确定性程序判分，Agent 只能分析错因和提交画像更新建议；
8. 副线表现只能作为待验证信号，不能直接认定为 CET-6 能力提升；
9. Prompt、Skill、上下文、工具调用、证据和最终决策之间的可追溯审计链；
10. 至少两个由模型真实自主调用的 Function Calling 工具。

本轮只完成架构分析和架构文档，不实现 Agent Runtime，不修改业务代码，不扩充数据库，不提前进入具体页面开发。

架构文档应说明设计选择、替代方案、边界、不变量、风险和验收方法，避免只堆模块名称。

## Prompt 3：阶段实施规划与代码开发

请先完整阅读：

* `AGENTS.md`
* `docs/REQUIREMENTS.md`
* `docs/ARCHITECTURE.md`
* 当前阶段的 `docs/PLAN.md`
* 当前代码、测试和 Git 状态

严格遵守已经确认的需求和架构，不得在实现过程中擅自改变系统架构、公共接口、核心数据模型、Agent 与确定性程序边界或 MVP 范围。

先根据当前仓库的真实状态生成本阶段实施计划。计划只覆盖一个规模适中的垂直切片，不要机械复制历史任务，也不要一次规划整个项目。

每个任务必须写清：

* 任务目标；
* 前置依赖；
* 允许修改的文件或目录；
* 禁止修改范围；
* 输入与输出；
* 逐步实现方式；
* 必须执行的命令；
* 测试要求；
* 验收标准；
* 失败时的回退方式。

计划确认后，按照任务顺序直接实现代码。实现时遵守以下规则：

1. 先理解现有代码，不重复实现已有能力；
2. 优先完成最小、可运行、可测试的实现；
3. 不用模拟结果冒充真实 Agent 或 Function Calling；
4. 不把客观题判分交给模型；
5. 不将隔离题内容泄漏进普通训练上下文；
6. 不信任前端传入的用户身份和权限字段；
7. 错误响应不得泄漏 API Key、数据库路径、SQL、traceback 或内部异常；
8. 必须补充或更新相应测试；
9. 完成后实际运行后端测试、前端构建和必要的浏览器验证；
10. 更新当前阶段计划状态，并说明完成内容、测试结果、风险和建议的中文 Git 提交信息。

不要只汇报"代码看起来正确"，必须用实际运行结果进行验证。

## Prompt 4：项目最终审查与收尾修复

你现在负责对 LingoForge 项目进行最终收尾审查和必要修复。

请先完整阅读：

* `AGENTS.md`
* README
* 需求文档
* 架构文档
* Agent 内部架构文档
* 当前实施计划
* 前后端源码
* 测试
* 配置文件
* 数据库初始化与重置脚本
* Git 状态

不要只根据文档中的"已完成"判断项目状态，必须检查代码和实际运行行为。

第一阶段，审查需求、架构、代码和实际行为是否一致，重点检查：

1. 核心用户流程能否真实走通；
2. Agent 是否真实进行决策和 Function Calling；
3. Agent、Skill、Workflow 和确定性程序的边界是否正确；
4. 客观题是否由程序判分；
5. 用户身份、数据隔离和隔离题防泄漏是否可靠；
6. 前后端接口字段、状态和错误处理是否一致；
7. DeepSeek 与 Mock 模式是否都符合项目约定；
8. README 中的安装、启动、测试和重置命令是否真实可用；
9. `.env.example` 是否完整，仓库是否存在密钥泄漏；
10. 核心开发 Prompt、运行时 Prompt、Skill、架构和数据模型是否可以在仓库中查看。

第二阶段，将发现的问题划分为：

* Blocking：影响安装、启动、测试、主流程或作业要求；
* Important：功能可运行但存在明显逻辑、边界或一致性问题；
* Optional：不影响本次交付的改进项。

第三阶段，直接修复全部 Blocking 问题，并修复合理范围内的 Important 问题。不要新增功能，不扩大 MVP，不进行与本次交付无关的大规模重构。

第四阶段，实际执行并记录：

* 后端测试；
* 前端构建；
* README 中的启动命令；
* Demo 数据重置；
* DeepSeek 配置验证；
* 浏览器核心流程；
* `git status` 和最终 diff 检查。

最后输出：

* 发现的问题；
* 已完成的修复；
* 实际执行的命令和结果；
* 仍然存在的限制；
* 当前是否达到可提交状态；
* 建议的中文 Git 提交信息。
